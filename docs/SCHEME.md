# CALTopo/SARTopo Integration — Plugin Design Scheme

## Overview

This document describes the design for adding CALTopo/SARTopo live-map integration to the CAP Field Plugin. The core capability is:

> A field operator opens the plugin, scans a CALTopo map-sharing QR code generated by the command post, and from that point the plugin automatically sends PLI (Position Location Information) and optionally other map data to the shared CALTopo/SARTopo map in real time.

No manual credential entry. No separate app. One scan connects the EUD.

---

## User Flow

```
[CMD Post generates QR from CALTopo map share screen]
         |
         v
[Field operator opens CAP plugin → taps "Scan Map QR"]
         |
         v
[Camera opens; operator scans QR code]
         |
         v
[Plugin parses connect key + server URL from QR]
         |
         v
[Plugin confirms: "Connected to Operation ALPHA map — tracking active"]
         |
         v
[Plugin continuously reports PLI to CALTopo map]
         |
         v
[Operator can optionally push markers / shapes / notes from ATAK]
```

## CALTopo QR Code Format

CALTopo map-sharing QR codes encode a URL in the following form:

```
https://caltopo.com/m/{MAP_ID}?key={CONNECT_KEY}
  -- or for SARTopo hosted instances --
https://{SARTOPO_HOST}/m/{MAP_ID}?key={CONNECT_KEY}
```

The plugin must handle both `caltopo.com` canonical URLs and self-hosted SARTopo instances (e.g., `sartopo.local`, `192.168.x.x`).

Extracted fields:

| Field | Source | Example |
|-------|--------|---------|
| `serverBase` | URL scheme + host | `https://caltopo.com` |
| `mapId` | URL path segment after `/m/` | `A1B2C` |
| `connectKey` | Query param `key` | `abc123xyz` |

The connect key alone is sufficient for position reporting. Full CRUD (adding markers, shapes) requires a credential ID + API key, which is a separate optional configuration step described in [Advanced Auth](#advanced-auth-team-api-credentials).

## Architecture

### Component Map

```
PluginTemplate (IPlugin)
    └── TopoMapPane (UI / Pane)
            ├── QRScanActivity (camera + ML Kit barcode)
            ├── TopoConnectionManager (holds parsed session, drives reporting)
            │       ├── PLIReporter (periodic ATAK GPS → CALTopo position API)
            │       ├── MarkerPusher (on-demand ATAK marker → CALTopo Marker API)
            │       └── CalTopoApiClient (HTTP + optional HMAC signing)
            └── TopoSessionStore (SharedPreferences — persists connect key across restarts)
```

### Key Classes to Create

| Class | Package | Purpose |
|-------|---------|---------|
| `TopoMapDropDown` | `...plugintemplate.topo` | Replaces bare `Pane`; owns lifecycle, delegates to sub-components |
| `QRScanActivity` | `...plugintemplate.topo` | Thin Activity wrapping ML Kit scan; returns parsed URL via Intent result |
| `CalTopoSession` | `...plugintemplate.topo.model` | Immutable value object: serverBase, mapId, connectKey, credential (nullable) |
| `TopoConnectionManager` | `...plugintemplate.topo` | Starts/stops background reporting thread; owns `CalTopoSession` |
| `PLIReporter` | `...plugintemplate.topo` | Reads ATAK self-location, POSTs to position API on interval |
| `CalTopoApiClient` | `...plugintemplate.topo.net` | Wraps `HttpURLConnection`; handles HMAC signing when credentials present |
| `TopoSessionStore` | `...plugintemplate.topo` | Persist/restore `CalTopoSession` via `SharedPreferences` |
| `MarkerPusher` | `...plugintemplate.topo` | On-demand POST of an ATAK `MapItem` as a CALTopo Marker |

## QR Scanning

### Library

Use **Google ML Kit Barcode Scanning** (`com.google.mlkit:barcode-scanning`). It is lightweight, offline-capable, and does not require Play Services on Android Go / MANET-isolated devices if the standalone model is bundled.

Alternative: **ZXing** (`com.journeyapps:zxing-android-embedded`) — simpler integration, no model file, but slightly less accurate in poor field lighting.

Recommendation: **ML Kit** for better reliability under field conditions (sun glare, damaged QR codes).

### Scanning Flow

1. User taps **"Scan Map QR"** button in `TopoMapPane`.
2. Plugin launches `QRScanActivity` via `startActivityForResult`.
   - `QRScanActivity` uses `CameraX` + ML Kit `BarcodeScanning` to process frames.
   - On successful decode: validates the URL matches the expected CALTopo pattern, returns result via `Intent`.
   - On cancel/failure: returns `RESULT_CANCELED`.
3. `TopoMapPane.onActivityResult` receives the URL string, calls `parseCalTopoUrl()`, constructs a `CalTopoSession`, and hands it to `TopoConnectionManager`.

### URL Parsing (pseudocode)

```java
CalTopoSession parseCalTopoUrl(String rawUrl) {
    Uri uri = Uri.parse(rawUrl);
    String host = uri.getScheme() + "://" + uri.getHost();
    List<String> segments = uri.getPathSegments(); // ["m", "A1B2C"]
    if (segments.size() < 2 || !segments.get(0).equals("m"))
        throw new IllegalArgumentException("Not a CALTopo map URL");
    String mapId = segments.get(1);
    String connectKey = uri.getQueryParameter("key");
    if (connectKey == null)
        throw new IllegalArgumentException("No connect key in URL");
    return new CalTopoSession(host, mapId, connectKey, null);
}
```

```xml

```

## PLI Reporting

### ATAK Self-Location API

ATAK exposes the device's current GPS fix through the `MapView` singleton and the self marker:

```java
MapView mapView = MapView.getMapView();
MapItem self = mapView.getSelfMarker();
GeoPoint location = self.getPoint();
double lat = location.getLatitude();
double lng = location.getLongitude();
double altMsl = location.getAltitude(); // meters MSL
String callsign = self.getMetaString("callsign", "UNKNOWN");
```

### CALTopo Position Report Endpoint

```
GET {serverBase}/api/v1/position/report/{connectKey}
    ?id={deviceId}
    &lat={latitude}
    &lng={longitude}
    &time={isoTimestamp}      (optional, ISO-8601)
    &accuracy={meters}        (optional)
```

No authentication header required when using a connect key — the key IS the auth token for position reports.

`deviceId` should be stable and human-readable. Use the ATAK callsign + UID:

```java
String deviceId = callsign + "_" + mapView.getSelfMarker().getUID();
```

### Reporting Interval

Default: **15 seconds**. User-configurable in plugin settings (10 s / 15 s / 30 s / 60 s / off).

Reporting runs on a `ScheduledExecutorService` background thread, not the UI thread.

### Battery / Network Awareness

- Pause reporting when ATAK is backgrounded (register `ActivityLifecycleCallbacks`).
- Backoff exponentially on consecutive HTTP errors (max 5 min).
- Log failures to a local ring buffer (last 20 errors) accessible in the plugin UI.

## CALTopo API Client

### HTTP Client

Use `java.net.HttpURLConnection` directly — no OkHttp dependency needed for simple GET/POST. The `build.gradle` currently excludes OkHttp from the implementation configuration anyway.

### Request Structure

```java
URL url = new URL(endpoint);
HttpURLConnection conn = (HttpURLConnection) url.openConnection();
conn.setRequestMethod("GET");
conn.setConnectTimeout(8_000);
conn.setReadTimeout(8_000);
conn.setRequestProperty("User-Agent", "CAP-ATAK-Plugin/1.0");
int code = conn.getResponseCode();
```

### Error Handling

| HTTP Code | Meaning | Action |
|-----------|---------|--------|
| 200 | Success | Clear error state |
| 401 / 403 | Bad connect key | Show "Connection rejected" in UI, stop reporting |
| 404 | Map not found | Show "Map not found" in UI, stop reporting |
| 429 | Rate limited | Backoff 60 s |
| 5xx | Server error | Exponential backoff |
| IOException | No network | Retry on next scheduled interval |

## Advanced Auth: Team API Credentials

For operators who want to push **markers, shapes, or track lines** (not just position pings), a full credential is required:

- **Credential ID** (12 chars)
- **API Key** (44 chars, Base64-encoded secret)

These are entered manually in plugin settings once per device and stored in `EncryptedSharedPreferences` (Android Jetpack Security).

### HMAC-SHA256 Request Signing

```java
// Build the string to sign
String expires = String.valueOf(System.currentTimeMillis() + 120_000); // 2 min from now
String toSign = method + " " + uri + "\n" + expires + "\n" + body;

// Sign it
byte[] keyBytes = Base64.decode(apiKey, Base64.DEFAULT);
Mac mac = Mac.getInstance("HmacSHA256");
mac.init(new SecretKeySpec(keyBytes, "HmacSHA256"));
String signature = Base64.encodeToString(mac.doFinal(toSign.getBytes(StandardCharsets.UTF_8)), Base64.NO_WRAP);

// Add headers
conn.setRequestProperty("Authorization", "HMAC " + credentialId + ":" + signature);
conn.setRequestProperty("X-Expires", expires);
```

## Data Types the User Can Send

The plugin UI exposes toggles for each data type. PLI is always on when connected; others are opt-in.

| Data Type | CALTopo Object | Trigger | Endpoint |
|-----------|---------------|---------|----------|
| PLI / Position | Position Report | Automatic (interval) | `/api/v1/position/report/{key}` |
| ATAK Marker | `Marker` | Manual — "Push to Map" button, or auto-push new markers | `/api/v1/map/{mapId}/Marker` |
| Track / Route | `LineString` | Manual export of ATAK track log | `/api/v1/map/{mapId}/LineString` |
| Search Area | `Polygon` | Manual — push selected polygon from ATAK | `/api/v1/map/{mapId}/Polygon` |

| Key | Storage | Value |

### How ATAK Iconsets Work

An ATAK iconset is a `.zip` archive containing:
- `iconset.xml` — metadata (name, UID, group, icon list)
- `{Group Name}/{icon}.png` — the actual PNG files

ATAK discovers iconsets by scanning the `iconsets/` directory inside the ATAK data folder on the device:

```
/sdcard/atak/iconsets/Incident Icons.zip
```

When the zip is present there, ATAK imports it into its internal `IconsetDatabase` and the icons become available in the marker picker under the group name.

---

### Bundling the Iconset in the Plugin

**Step 1 — Add the zip to plugin assets**

Place `Incident Icons.zip` (the original WinTAK export, unchanged) at:

```
app/src/main/assets/iconsets/Incident Icons.zip
```

**Step 2 — Install on plugin start**

In `PluginTemplate.onStart()` (or in `TopoMapDropDown` initialization), copy the zip to the ATAK iconsets directory if it is not already there:

```java
private void installIconset() {
    File atakIconsetsDir = new File(
        Environment.getExternalStorageDirectory(), "atak/iconsets");
    atakIconsetsDir.mkdirs();

    File dest = new File(atakIconsetsDir, "Incident Icons.zip");
    if (dest.exists()) return; // already installed

    try (InputStream in = pluginContext.getAssets().open("iconsets/Incident Icons.zip");
         FileOutputStream out = new FileOutputStream(dest)) {
        byte[] buf = new byte[4096];
        int len;
        while ((len = in.read(buf)) > 0) out.write(buf, 0, len);
    } catch (IOException e) {
        Log.e(TAG, "Failed to install Incident Icons iconset", e);
        return;
    }

    // Tell ATAK to reload iconsets so the new one is available immediately
    AtakBroadcast.getInstance().sendBroadcast(
        new Intent("com.atakmap.app.REFRESH_ICONSET"));
}
```

Call `installIconset()` from `onStart()`. The existence check on `dest` makes it a no-op after the first install, so it adds no meaningful overhead on subsequent starts.

> **Storage permission:** The ATAK data directory is on external storage. The plugin already runs inside the ATAK process which holds `WRITE_EXTERNAL_STORAGE`, so no additional permission declaration is needed.

**Step 3 — Version / update handling**

If the iconset ever changes (new icons, revised artwork), bump the version by writing the new zip over the existing file. The simplest approach is to store the current iconset version in `SharedPreferences` and overwrite the zip when the stored version is older than the bundled version:

```java
int BUNDLED_VERSION = 1; // increment when iconset changes
int installedVersion = prefs.getInt("iconset_version", 0);
if (!dest.exists() || installedVersion < BUNDLED_VERSION) {
    // copy zip, then:
    prefs.edit().putInt("iconset_version", BUNDLED_VERSION).apply();
    AtakBroadcast.getInstance().sendBroadcast(
        new Intent("com.atakmap.app.REFRESH_ICONSET"));
}
```

## In-Plugin Icon Dropper

### Goal

Field operators can place any CAP incident icon directly from the plugin pane without navigating to ATAK's built-in icon dropper. Tapping an icon in the plugin grid initiates a placement flow and drops the marker on the map.

---

### Placement Modes

Three placement modes are supported. The user picks the default in Settings; a long-press on any icon overrides it for that single drop.

| Mode | Behavior | Best For |
|------|----------|----------|
| **Tap to Place** | Plugin pane minimizes; user taps any point on the map; marker drops there | Precise placement away from self |
| **At My Position** | Marker drops immediately at the device's current GPS fix | Speed — marking own location |
| **At Map Center** | Marker drops at the center of the current map view | When the map is already panned to the target |

Default mode: **Tap to Place** (most useful in the field; matches the mental model of the native dropper).

### Tap-to-Place Flow

This mode requires dismissing the plugin pane so the user can interact with the map. ATAK provides a mechanism for exactly this via `ToolListener` / `MapTouchController`.

**Step-by-step:**

1. User taps an icon cell in the grid.
2. Plugin records the selected icon path and label in `PendingDrop` (a simple holder object).
3. Plugin calls `MapView.getMapView().getMapTouchController().setToolActive(iconDropTool)` to activate a custom `Tool` subclass (`IconDropTool`).
4. `IconDropTool.onToolBegin()` collapses/hides the plugin pane via `uiService.hidePane(templatePane)` and posts a toast: _"Tap map to place [icon name]"_.
5. User taps the map. `IconDropTool.onMapEvent(MapEvent)` intercepts the tap, extracts `GeoPoint` from the event.
6. `IconDropTool` calls `IconDropper.place(iconPath, label, geoPoint)` (described below).
7. `IconDropTool.onToolEnd()` restores the plugin pane and clears `PendingDrop`.
8. If the user presses Back before tapping, the Tool is deactivated with no marker placed.

### Marker Placement API

`IconDropper` is a single static-method utility that creates and registers the `PlacemarkMapItem`:

```java
public final class IconDropper {

    public static PlacemarkMapItem place(String iconPath, String title, GeoPoint point) {
        MapView mv = MapView.getMapView();

        PlacemarkMapItem marker = new PlacemarkMapItem(
            UUID.randomUUID().toString(),
            "a-f-G-E-V-A");   // CoT type: friendly-ground-equipment-vehicle-air
                               // Override per icon type if needed (see CoT Type Map below)

        marker.setTitle(title);
        marker.setPoint(point);
        marker.setIcon(new Icon(iconPath));
        marker.setMetaString("how", "h-g-i-g-o"); // how=human-GPS-input-gesture-other
        marker.setMetaBoolean("archive", true);    // persist across ATAK restarts
        marker.setMetaBoolean("editable", true);
        marker.setMetaString("menu", "menus/default.xml");

        mv.getRootGroup().addItem(marker);

        // Bring the marker into view briefly
        mv.getMapController().panTo(point, true);

        return marker;
    }
}
```

### Loading Icon Images in the Grid

The PNGs are inside the zip on the device at `/sdcard/atak/iconsets/Incident Icons.zip`. Load them via `ZipFile` into `Bitmap`s for the grid:

```java
// Called once; results cached in a Map<String, Bitmap>
Map<String, Bitmap> loadIconBitmaps() throws IOException {
    Map<String, Bitmap> bitmaps = new LinkedHashMap<>();
    File zip = new File(
        Environment.getExternalStorageDirectory(), "atak/iconsets/Incident Icons.zip");
    try (ZipFile zf = new ZipFile(zip)) {
        Enumeration<? extends ZipEntry> entries = zf.entries();
        while (entries.hasMoreElements()) {
            ZipEntry entry = entries.nextElement();
            if (entry.getName().endsWith(".png")) {
                try (InputStream is = zf.getInputStream(entry)) {
                    Bitmap bmp = BitmapFactory.decodeStream(is);
                    // key is just the filename, e.g. "Command Post.png"
                    String name = new File(entry.getName()).getName();
                    bitmaps.put(name, bmp);
                }
            }
        }
    }
    return bitmaps;
}
```

Load this off the UI thread (e.g., in a `ViewModel` or background thread) and post the result to the adapter. Cache the map for the plugin session lifetime; invalidate only if the iconset version changes.

## ArcGIS Quick Capture Integration

### What Is Quick Capture

ArcGIS Quick Capture is an Esri field data collection app designed for rapid one-tap record submission. CAP units use it for structured reporting — clue logs, asset reports, incident forms — where each submission writes a row into an ArcGIS Feature Service. The plugin integration replaces the two-step workflow of (1) switching apps to open Quick Capture then (2) manually entering location with a single QR scan that lands the user directly in the right project with location already filled in.

---

### Quick Capture Share Link / QR Format

Quick Capture projects are shared via a URL in one of two forms:

```
# HTTPS deep link (preferred — works on both iOS and Android)
https://quickcapture.arcgis.app/?itemID={PROJECT_ITEM_ID}

# Custom URL scheme (also supported; harder to scan on Android)
arcgis-quickcapture://?itemID={PROJECT_ITEM_ID}
```

For enterprise (non-ArcGIS Online) portals, a `portalUrl` parameter is appended:

```
https://quickcapture.arcgis.app/?itemID={PROJECT_ITEM_ID}&portalUrl=https://gis.example.gov/portal
```

The QR code encodes one of these URLs verbatim. Scanning it with the plugin extracts:

| Field | Source | Example |
|-------|--------|---------|
| `itemId` | Query param `itemID` | `aabda4a5e36d42c2bcf1c479fe01e5e3` |
| `portalUrl` | Query param `portalUrl` (nullable) | `https://gis.cap.gov/portal` |
| `scheme` | URL scheme — `https` or `arcgis-quickcapture` | Used to detect QC links vs CALTopo links |

---

### Implementation Approach: Direct Feature Service Submission

The plugin submits records directly to the ArcGIS Feature Service REST API that backs the Quick Capture project — no Quick Capture app installed or opened. The user scans the QR once to identify the project, the plugin resolves the Feature Service endpoint, and all subsequent submissions happen inside the plugin with location pre-filled from live ATAK PLI.

> **Tier 1 (launching the Quick Capture app via deep link) is deferred.** The QC URL scheme's `center` parameter only repositions the map view and cannot inject GPS into device variables, so Tier 1 offers minimal location pre-fill benefit over the native QC workflow. Tier 2 gives full control over every field.

---

### Step 1: Resolve Feature Service URL from Item ID

The QR code only provides an `itemID`. Before the first submission the plugin must resolve this to an actual Feature Service endpoint. This is a one-time network call whose result is cached in `QcSessionStore`.

**ArcGIS item data endpoint:**
```
GET {portalBase}/sharing/rest/content/items/{itemID}/data
    ?f=json
    &token={apiKey}
```

For ArcGIS Online the portal base is `https://www.arcgis.com`.  
For enterprise it is the `portalUrl` from the QR.

**Project JSON structure (abbreviated):**
```json
{
  "layers": [
    {
      "id": 0,
      "url": "https://services.arcgis.com/xyz/arcgis/rest/services/CAPAssetReport/FeatureServer/0",
      "fields": [
        { "name": "callsign",          "type": "esriFieldTypeString" },
        { "name": "team",              "type": "esriFieldTypeString" },
        { "name": "latitude",          "type": "esriFieldTypeDouble" },
        { "name": "longitude",         "type": "esriFieldTypeDouble" },
        { "name": "altitude",          "type": "esriFieldTypeDouble" },
        { "name": "horizontal_accuracy","type": "esriFieldTypeDouble" },
        { "name": "capture_time",      "type": "esriFieldTypeDate"   },
        { "name": "notes",             "type": "esriFieldTypeString" }
      ]
    }
  ]
}
```

The plugin parses `layers[0].url` as the Feature Service endpoint and `layers[0].fields` to know which fields exist and their types. This drives the dynamic form (see UI section). All layers beyond index 0 are ignored unless the project defines multiple capture buttons — multi-layer support is deferred.

**Resolution flow:**

```
QR scanned → itemID extracted
     │
     ├─ QcSessionStore has cached URL for this itemID?
     │       └─ YES → skip fetch, use cached
     │
     └─ NO → fetch /sharing/rest/content/items/{itemID}/data
                  │
                  ├─ 200 OK → parse layers[0].url + fields → cache → show form
                  ├─ 401/403 → "Authentication failed — check API key in Settings"
                  └─ IOException → "Cannot reach ArcGIS — check network"
```

If the device is offline and no cached URL exists, the plugin prompts: _"Cannot connect to ArcGIS to set up this project. Connect to a network once, then you can submit offline."_ After the first successful resolution the form works fully offline (queue path).

---

### Step 2: Feature Service Submission

**Endpoint:**
```
POST {featureServiceUrl}/addFeatures
```

**Request:** `Content-Type: application/x-www-form-urlencoded`

```
features=[
  {
    "attributes": {
      "callsign":           "ALPHA-7",
      "team":               "Bravo",
      "latitude":           38.9072,
      "longitude":          -77.0369,
      "altitude":           85.3,
      "horizontal_accuracy": 4.1,
      "capture_time":       1748700120000,
      "notes":              "Clue near treeline"
    },
    "geometry": {
      "x": -77.0369,
      "y": 38.9072,
      "spatialReference": { "wkid": 4326 }
    }
  }
]
token={API_KEY}
f=json
```

**Successful response:**
```json
{
  "addResults": [
    { "objectId": 1042, "globalId": "{...}", "success": true }
  ]
}
```

**Error responses to handle:**

| Condition | Response | Plugin Action |
|-----------|----------|---------------|
| Bad token | HTTP 499 or `{"error":{"code":498}}` | Clear token, prompt re-entry |
| Field missing/wrong type | `success: false` with error message | Show field-level error in form |
| Service unavailable | HTTP 5xx | Queue for offline retry |
| No network | `IOException` | Queue for offline retry |
| Success | HTTP 200, `success: true` | Flash green, show objectId, reset form |

**Feature Service add endpoint:**

```
POST https://services.arcgis.com/{serviceId}/arcgis/rest/services/{serviceName}/FeatureServer/{layerId}/addFeatures
```

or for enterprise:

```
POST {portalUrl}/server/rest/services/{serviceName}/FeatureServer/{layerId}/addFeatures
```

**Request body (form-encoded):**

```
features=[{"attributes":{"callsign":"ALPHA-7","latitude":38.9072,"longitude":-77.0369,...},"geometry":{"x":-77.0369,"y":38.9072}}]
token={ACCESS_TOKEN}
f=json
```

**ATAK PLI → Feature Service field mapping:**

| ATAK Source | Feature Service Field (example) | Type |
|-------------|--------------------------------|------|
| `self.getPoint().getLatitude()` | `latitude` | Double |
| `self.getPoint().getLongitude()` | `longitude` | Double |
| `self.getPoint().getAltitude()` | `altitude` | Double |
| GPS accuracy from `LocationManager` | `horizontalAccuracy` | Double |
| `System.currentTimeMillis()` | `captureTime` | Date (epoch ms) |
| `self.getMetaString("callsign","")` | `callsign` | String |
| `self.getMetaString("team","")` | `team` | String |
| User-entered form fields | any additional fields | varies |

Field names are project-specific. The plugin Settings screen lets admins configure the mapping once.

**Geometry** (required for point features):

```json
{
  "x": -77.0369,
  "y": 38.9072,
  "spatialReference": { "wkid": 4326 }
}
```

**v1: ArcGIS API Key (implemented first)**

A single static string appended as a `token` query parameter to every request. No OAuth flow, no token refresh, appropriate for CAP organizational Feature Services. Stored in `EncryptedSharedPreferences`. Entered once in plugin Settings.

**v2: Portal token / OAuth (deferred)**

For enterprise portals that do not accept API keys, a portal-generated token via `generateToken` is needed. Implement after v1 is stable. The `ArcGisAuthManager` is already the single auth seam — swap the implementation there without touching call sites.

### QR Detection: Distinguishing Quick Capture from CALTopo

The existing QR scan entry point is shared. After scanning, the plugin inspects the URL to route to the correct handler:

```java
ScanResult classifyQrUrl(String rawUrl) {
    Uri uri = Uri.parse(rawUrl);
    String host = uri.getHost();
    String scheme = uri.getScheme();

    if ("quickcapture.arcgis.app".equals(host)
            || "arcgis-quickcapture".equals(scheme)) {
        return ScanResult.QUICK_CAPTURE;
    }
    if (uri.getQueryParameter("key") != null
            && uri.getPathSegments().contains("m")) {
        return ScanResult.CALTOPO;
    }
    return ScanResult.UNKNOWN;
}
```

The plugin UI then routes `QUICK_CAPTURE` to `QuickCaptureDropDown` and `CALTOPO` to `TopoMapDropDown`.

### ATAK Field Resolver

```java
String resolveAtakField(String fieldName, String callsignOverride) {
    MapView mv = MapView.getMapView();
    Marker self = mv.getSelfMarker();
    switch (fieldName) {
        case "callsign":  return callsignOverride != null
                                ? callsignOverride
                                : self.getMetaString("callsign", "");
        case "uid":       return self.getUID();
        case "team":      return self.getMetaString("team", "");
        case "timestamp": return new SimpleDateFormat("yyyy-MM-dd'T'HH:mm:ss'Z'",
                                Locale.US).format(new Date());
        case "lat":       return String.valueOf(self.getPoint().getLatitude());
        case "lng":       return String.valueOf(self.getPoint().getLongitude());
        case "alt":       return String.valueOf(self.getPoint().getAltitude());
        default:          return null;
    }
}
```

### Offline Handling

Quick Capture's native app queues records when offline and syncs on reconnect. The plugin's direct Tier 2 path must replicate this for disconnected environments:

- On `IOException` / no network: write the submission payload to a local queue (JSON file in app-private storage).
- Background service (or a `WorkManager` job) retries the queue on network availability.
- Queue is displayed in the plugin UI as "N records pending sync."
- Records in the queue include a full snapshot of the PLI at time of capture so location doesn't drift.

### GeoChat CoT Construction

**Event UID format** (matches GeoChat convention):
```
GeoChat.{senderUID}.{chatroomId}.{randomUUID}
```

**For a direct message to one contact:**

```java
CotEvent buildChatEvent(String message, String senderUid, String senderCallsign,
                         String recipientUid, String recipientCallsign) {
    String eventUid = "GeoChat." + senderUid + "." + recipientCallsign
                    + "." + UUID.randomUUID();
    long now = System.currentTimeMillis();

    CotEvent event = new CotEvent();
    event.setUID(eventUid);
    event.setType("b-t-f");
    event.setTime(new CoTIso8601Time(now));
    event.setStart(new CoTIso8601Time(now));
    event.setStale(new CoTIso8601Time(now + 60_000));
    event.setHow("h-g-i-g-o");

    // Point is conventional 0,0 for chat
    CotPoint point = new CotPoint(0.0, 0.0, 9999999.0, 9999999.0, 9999999.0);
    event.setPoint(point);

    CotDetail detail = new CotDetail();

    // __chat element
    CotDetail chat = new CotDetail("__chat");
    chat.setAttribute("id",             recipientUid);
    chat.setAttribute("chatroom",       recipientCallsign);
    chat.setAttribute("groupOwner",     "false");
    chat.setAttribute("senderCallsign", senderCallsign);
    CotDetail chatGrp = new CotDetail("chatgrp");
    chatGrp.setAttribute("id",   recipientUid);
    chatGrp.setAttribute("uid0", senderUid);
    chatGrp.setAttribute("uid1", recipientUid);
    chat.addChild(chatGrp);
    detail.addChild(chat);

    // link element
    CotDetail link = new CotDetail("link");
    link.setAttribute("uid",      senderUid);
    link.setAttribute("type",     "a-f-G-U-C");
    link.setAttribute("relation", "p-p");
    detail.addChild(link);

    // remarks (message body)
    CotDetail remarks = new CotDetail("remarks");
    remarks.setAttribute("source", "BAO.F.ATAK." + senderUid);
    remarks.setAttribute("to",     recipientUid);
    remarks.setAttribute("time",   new CoTIso8601Time(now).toString());
    remarks.setInnerText(message);
    detail.addChild(remarks);

    // marti routing
    CotDetail marti = new CotDetail("marti");
    CotDetail dest  = new CotDetail("dest");
    dest.setAttribute("callsign", recipientCallsign);
    marti.addChild(dest);
    detail.addChild(marti);

    event.setDetail(detail);
    return event;
}
```

**For a group / broadcast:**
- Set `chatroom` and `id` to the group name (e.g. `"All Chat Rooms"`)
- Add one `<chatgrp>` per group member
- Add one `<dest callsign="..."/>` per member inside `<marti>`
- Use `dispatcher.dispatchToBroadcast(event)` instead of `dispatchToContact`

**Dispatch:**

```java
CotDispatcher dispatcher = CotMapComponent.getExternalDispatcher();

for (ChatTarget target : selectedTargets) {
    CotEvent event = buildChatEvent(
        renderedMessage,
        MapView.getMapView().getSelfMarker().getUID(),
        selfCallsign,
        target.uid,
        target.callsign
    );
    if (target.isBroadcast)
        dispatcher.dispatchToBroadcast(event);
    else
        dispatcher.dispatchToContact(event, target.uid);
}
```

### Template Persistence

Templates are stored as a JSON array in `SharedPreferences`:

```json
[
  {
    "id": "builtin_ops_status",
    "name": "Ops Status",
    "template": "{callsign}, Ops {status}, Task {task}, {lat_dms} {lng_dms}, Bat {battery}%",
    "builtin": true,
    "lastRecipients": ["uid-of-bravo3", "uid-of-ops-group"]
  },
  {
    "id": "custom_abc123",
    "name": "My Custom Template",
    "template": "{callsign} at {mgrs} — {note}",
    "builtin": false,
    "lastRecipients": []
  }
]
```

Built-in templates (where `builtin: true`) cannot be deleted. Their `template` string can be overridden by storing a user-modified copy under the same `id`.
